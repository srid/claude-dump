## 2. [F3](https://github.com/future-file-format/f3) | [126 comments](https://news.ycombinator.com/item?id=48647799) | 575 pts

**Article:** F3 (Future-proof File Format) is a research-prototype columnar storage format positioned as a successor to Parquet and ORC, aiming to fix what its authors call the layout shortcomings of those decade-old formats while improving efficiency, interoperability, and extensibility. Its defining feature is that each self-describing file embeds not just data and metadata but also WebAssembly (Wasm) decoders, which add only kilobytes of overhead and let any platform read the file even when native decoders are unavailable. This design supports user-defined encoding schemes that can be added without rewriting the format. The repository notes it is a research prototype not intended for production use.

**Discussion:** Commenters criticized the README for failing to clearly explain what F3 is (a Parquet-style columnar format) or why one would switch, with several pointing to the academic paper for the motivation—chiefly that Parquet is hard to extend with new encodings while staying compatible with old readers, which embedded Wasm decoders aim to solve via forward compatibility. The embedded-Wasm-decoder idea drew the most debate: some called it genius for language-agnostic, future-proof decoding, while many raised security concerns about executing code from data files, countered by arguments that Wasm sandboxing has no I/O, supports memory limits and gas metering, and is far less risky than RCE. Skeptics questioned its practical value (decoded output is still format-specific, legacy hardware may be too slow, archival reliance on a future Wasm interpreter), and noted the repo has had no commits in eight months, suggesting limited adoption.

## 3. [Unlimited OCR: One-shot long-horizon parsing](https://github.com/baidu/Unlimited-OCR) | [96 comments](https://news.ycombinator.com/item?id=48643426) | 419 pts

**Article:** Unlimited-OCR is an open-source OCR model from Baidu, built on ideas from DeepSeek-OCR/DeepSeek-OCR-2 and PaddleOCR, designed for "one-shot long-horizon parsing" — transcribing long, multi-page documents in a single inference pass rather than page-by-page. Its core technique, Reference Sliding Window Attention (R-SWA), keeps full global attention on the source document image while restricting attention over the model's own generated text to a small moving window, avoiding the linearly growing KV cache that slows and inflates memory in standard LLM decoders. It offers a "Gundam" mode for single cropped images and a "Base" mode for multi-page PDFs, supports up to a 32,768-token context, and ships weights on Hugging Face/ModelScope with transformers and SGLang (streaming) inference.

**Discussion:** A central debate is whether OCR is "solved" by vision models: skeptics argue traditional OCR is faster, cheaper, and more reliable, while many practitioners counter that OCR remains unsolved—especially for layout, tables, non-English scripts (CJK, Arabic, Thai, diacritics), and complex documents—where vision-LLMs are a clear step forward despite harder-to-detect failure modes and hallucinated/translated artifacts. Several commenters note the R-SWA approach essentially formalizes the manual chunking/image-slicing they already do, and discuss tools they rely on (PaddleOCR, Mistral OCR, Claude, marker, Azure Document Intelligence, AWS Textract). Commenters praised Baidu for crediting DeepSeek-OCR and PaddleOCR, and a long tangent lamented the poor state of optical music recognition (OMR) and the lack of good training corpora and notation formats (MusicXML, Lilypond, ABC, MEI).

Commenters also raised memcached's own rough edges, notably slab allocation and "slab starvation," and weighed Redis/Valkey alternatives, with a tangent on whether an RDBMS or local in-process cache (e.g. APCu) makes a separate cache server unnecessary given network latency.

## 16. [The new HTTP QUERY method explained](https://kreya.app/blog/new-http-query-method-explained/) | [159 comments](https://news.ycombinator.com/item?id=48640974) | 208 pts

**Article:** The article explains HTTP QUERY, a new method (RFC 9110-era, defined in a dedicated RFC) that behaves like GET but officially permits a request body, aimed at complex search and filter queries that don't fit cleanly into URL parameters. It frames QUERY as solving the limitations of three existing approaches: GET with query strings (length limits, awkward encoding of nested data), GET with a body (inconsistently supported by browsers, proxies, and servers), and POST (non-idempotent semantics that block automatic retries and caching). QUERY is described as safe, idempotent, and cacheable—provided caches incorporate the request body into the cache key. The piece cautions that infrastructure support is still limited and that simple cases or shareable, link-able results are still better served by ordinary GET.

**Discussion:** A central debate is whether a whole new method is warranted versus simply fixing GET to allow bodies; proponents argue a new method gives intermediaries (proxies, CDNs, firewalls, caches) unambiguous signals that a request is safe, idempotent, and body-carrying, and that unpatched servers can cleanly return 405 and fall back to POST. Skeptics counter that QUERY just amounts to "GET with a body," that POST workarounds already work, and that real-world adoption will be slow because middleboxes, CDNs, and corporate proxies must all cooperate. Commenters cite concrete use cases (Elasticsearch/OpenSearch search, GraphQL, reverse image search) and note advantages over POST around cacheability—pointing out even Cloudflare lacks native POST caching. Several also flag that QUERY results aren't shareable via URL and question whether such a simple addition needs this much explanation.
