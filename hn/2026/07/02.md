## 8. [Since Linux 6.9, LUKS suspend stopped wiping disk-encryption keys from memory](https://mathstodon.xyz/@iblech/116769502749142438) | [176 comments](https://news.ycombinator.com/item?id=48763035) | 363 pts

**Article:** Ingo Blechschmidt recounts discovering that since Linux 6.9 (May 2024), the cryptsetup-suspend tool (originally pioneered by Debian) used to lock a LUKS full-disk-encrypted laptop on suspend had been silently failing: while it still prompted for the passphrase on resume, an extra copy of the volume/master key remained resident in kernel memory across suspend, leaving it extractable by anyone who seized the still-powered laptop. The piece reflects on the unease of trusting a security mechanism for years only to find it was never doing its job.

**Discussion:** Commenters clarified the mechanics — the bug was insidious precisely because users were still prompted for their passphrase on resume, making the setup appear secure while a stray key copy lingered in RAM — and debated how narrow the real-world threat is, since exploiting it requires physical access to a suspended machine's memory (e.g. seizure or cold-boot attacks). A large tangent compared full-disk-encryption options across platforms, weighing BitLocker (criticized as proprietary and for Microsoft escrowing recovery keys, defended as excellent for managing enterprise fleets, especially with a PIN), FileVault (noted to also have optional iCloud key escrow), VeraCrypt, and LUKS. Others discussed mitigations like hibernate-to-disk, hardware transparent memory encryption on recent Intel/AMD CPUs (with a subthread on AMD walking back plans to drop it from consumer chips), and TPM+PIN configurations. A final thread argued over whether the silent failure reflects Linux's decentralized, multi-distro development model versus single-tree OSes.
